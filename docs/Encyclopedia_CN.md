# DageChat 终极百科全书

## 第一章：核心理念与定位 (Philosophy & Positioning)

> *"In cryptography we trust. In math we trust. In central servers we do not."*
> — DageChat 宣言

### 1.1 黑暗森林假说与 DageChat 的生存论

**1.1.1 互联网的现状：全景监狱 (The Panopticon)**
当代的互联网并非一个自由的公共广场，而是一个经过精心设计的“全景监狱”。在这个监狱中，中心化服务提供商（ISP、社交平台巨头）占据了塔楼的中心位置，拥有无死角的“上帝视角”。
*   **透明的猎物**：用户在网络上的每一次点击、每一条消息、每一次连接，都在向中心服务器广播自己的坐标。
*   **被记录的历史**：即便内容被加密，元数据（Metadata）——即“谁在何时与谁进行了多久的通信”——依然被完整记录。这些元数据足以构建出精确的社会关系图谱，甚至预测用户的行为。

**1.1.2 黑暗森林法则的引入**
DageChat 将刘慈欣《三体》中的“黑暗森林法则”引入网络空间，作为其核心生存哲学的基石：
1.  **生存是第一需要**：在对抗性网络环境中，通讯的首要目标不是“好用”，而是“生存”（抗审查、抗封锁）。
2.  **他人即地狱**：网络中的节点（中继器、路由节点）应当被默认为恶意的或被监控的。我们不能假设基础设施是善意的。
3.  **暴露即毁灭**：一旦用户的真实身份（IP + 公钥）与敏感行为（群聊内容）建立关联，这种关联就是永久性的罪证。

**1.1.3 DageChat 的战术回应**
基于上述假设，DageChat 不试图去“感化”猎人（请求平台保护隐私），而是为用户提供了一套**“二向箔”级别的生存工具**：
*   **隐形（Stealth）**：通过 NIP-59 和幽灵群技术，将有意义的通讯伪装成背景噪音。
*   **去中心化（Decentralization）**：没有中心塔楼。猎人无法通过摧毁一个节点来消灭整个网络。
*   **无关性（Agnosticism）**：中继器不知道它在传输什么，甚至不知道它在为谁服务。

### 1.2 协议主权主义 vs. 平台封建主义

这是 DageChat 最核心的政治经济学主张。我们正在经历从 Web2 到 Web3 的范式转移，其实质是**数据所有权**的战争。

**1.2.1 平台封建主义 (Platform Feudalism)**
目前的互联网模型是封建制的：
*   **领主（平台）**：拥有服务器、数据库和算法。他们制定规则（EULA），拥有生杀大权（封号）。
*   **佃农（用户）**：在领主的土地上通过生产内容（聊天、发帖）来换取暂时的居住权。
*   **本质缺陷**：佃农不拥有土地。一旦被领主驱逐，佃农将失去所有社会关系（Social Graph）和历史记忆（Chat History）。**在这种模式下，隐私是领主的恩赐，而非用户的权利。**

**1.2.2 协议主权主义 (Protocol Sovereignty)**
DageChat 倡导“协议即法律，私钥即主权”。
*   **身份主权 (Identity Sovereignty)**：
    *   在 DageChat 中，身份不再是数据库中的一行记录，而是一个数学概念（`secp256k1` 公私钥对）。
    *   **无许可性 (Permissionless)**：生成一个身份不需要向腾讯、Telegram 或任何政府机构申请。只要你会抛硬币（生成随机数），你就拥有身份。这是一个**自然权利**。
*   **数据主权 (Data Sovereignty)**：
    *   **本地优先 (Local-First)**：数据的第一存储地是用户自己的硬盘（SQLite），而非云端。
    *   **持有权**：解密后的数据属于用户。即使 DageChat 项目方消失，即使所有 Relay 关闭，只要用户持有私钥和本地数据库，他的历史记忆就永远存在。
*   **网络主权 (Network Sovereignty)**：
    *   用户拥有“用脚投票”的权利。如果某个中继器开始审查言论，用户可以在毫秒级切换到另一个中继器，且不丢失任何好友关系或历史数据。

### 1.3 降维打击：与传统 IM 的深度对比

为了明确 DageChat 的生态位，我们将从架构、隐私和抗审查三个维度，将其与主流软件进行残酷的对比。

#### 1.3.1 对比 WeChat / QQ (完全中心化)
*   **架构**：WeChat 是典型的“老大哥”模式。服务端拥有密钥，可以查看所有聊天记录（用于审核），可以随时物理阻断账号。
*   **DageChat 的优势**：完全的端到端加密（E2EE），服务端无法解密；完全的去中心化，不存在“封号”概念（因为没有账号管理器）。

#### 1.3.2 对比 Telegram (混合型)
*   **架构**：Telegram 的普通群聊**不是**端到端加密的，数据明文存储在服务器上。虽然它号称不配合监管，但技术上它**有能力**出卖用户。且其账号依赖手机号（SIM 卡），这是现实世界身份的强绑定。
*   **DageChat 的优势**：
    *   DageChat 默认全加密（群聊、私聊）。
    *   无需手机号，彻底切断数字身份与物理身份的强制关联。
    *   不依赖 Durov 兄弟的个人道德，只依赖数学。

#### 1.3.3 对比 Signal (隐私标杆)
Signal 是目前 Web2 隐私通讯的黄金标准，但在极端对抗环境下仍有局限。
*   **局限性 1：元数据泄露**。Signal 知道“谁在和谁说话”（Sealed Sender 仅部分解决了此问题）。
*   **局限性 2：中心化服务**。Signal 依赖 AWS/Google 云服务。如果 Signal 服务器被国家级防火墙屏蔽，通讯即中断。Signal 也有中心化的手机号验证机制。
*   **DageChat 的优势**：
    *   **幽灵群 (Ghost Group)**：通过共享私钥技术，实现了 Signal 做不到的**“不可归因性”**（Unattributability）。在网络层面上，无法区分消息是由群内哪个成员发送的。
    *   **抗单点故障**：DageChat 不依赖单一服务商。只要世界上还有一台 Relay 在运行，网络就是通的。

### 1.4 目标用户群 (Target Audience)

DageChat 不是为所有人设计的。它牺牲了一定的便利性（如找回密码、云端同步、表情包商店），换取了极致的安全与主权。

**它是为以下人群设计的“诺亚方舟”：**
1.  **吹哨人与调查记者**：需要在一个绝对无法被追踪的环境中传输高敏信息。
2.  **加密货币持有者与 Cypherpunks**：信仰“私钥即一切”，拒绝任何形式的中心化托管。
3.  **受压迫地区的活动家**：面临极高的物理审查风险，需要一款无法被封锁的通讯工具。
4.  **企业核心决策层**：防止商业机密通过第三方服务器泄露（即使是 Slack 或 Teams 也有内部员工泄密的风险）。

## 本章问答 (Q&A)

**Q1: DageChat 所谓的“无许可性”究竟是什么意思？**
**A:** “无许可”意味着你不需要任何人的批准即可加入网络。你不需要验证手机号、不需要邮箱、不需要实名认证，甚至不需要通过验证码。你只需要在本地运行算法生成一对密钥，你就成为了网络的一部分。没有任何“管理员”可以阻止你生成密钥或禁止你使用该密钥签名。

**Q2: 既然没有中心服务器，如果我忘了密码（丢失了私钥），谁能帮我找回？**
**A:** **没有人。** 这是“主权”的代价。就像你丢失了家里的金条一样，没有银行会赔偿你。DageChat 赋予你对账号的绝对控制权，同时也要求你承担绝对的责任。请务必离线备份你的私钥（nsec）。

**Q3: 为什么说 Telegram 是不安全的，而 DageChat 是安全的？**
**A:** Telegram 的群聊默认不加密，且服务端拥有所有数据。Telegram 的安全性依赖于“相信杜洛夫不作恶”。而 DageChat 的群聊默认使用 XChaCha20 加密，且服务端（Relay）根本拿不到解密密钥。DageChat 的安全性依赖于“数学无法被攻破”。在信任模型上，数学优于人性。

**Q4: “黑暗森林”比喻听起来很可怕，这是否意味着 DageChat 充斥着非法活动？**
**A:** 技术是中立的。“黑暗森林”描述的是**网络环境的险恶**（无处不在的监控和审查），而不是用户行为。DageChat 就像一把加密的盾牌，既可以保护自由斗士，也可能被坏人利用。但作为基础设施，我们认为保护通信自由的权利高于一切，就像公路建设者不负责检查车里装的是什么。

**Q5: 如果政府封锁了所有的 DageChat 中继器（Relay）怎么办？**
**A:** 这是不可能的。Relay 协议非常轻量级（基于 WebSocket），任何人都可以在几分钟内用一台 VPS 甚至树莓派搭建一个 Relay。封锁一个 Relay，会有十个新的冒出来。此外，DageChat 支持 Tor 代理，可以连接暗网中的 Onion Relay，实现物理层面的抗封锁。

**Q6: DageChat 的“幽灵群”和 Signal 的“群组”有什么本质区别？**
**A:** 在 Signal 群组中，服务器知道群里有哪些成员，且每条消息都带有发送者的签名，也就是说，虽然内容加密了，但**“Alice 发了一条消息”**这个事实是确定的。在 DageChat 幽灵群中，所有成员共享同一个“群私钥”发消息。对于外界（包括服务器），看起来就像是**“群自己对自己说了一句话”**，根本无法分辨是 Alice 说的还是 Bob 说的。这叫“不可归因性”。

**Q7: 我能用 DageChat 给微信好友发消息吗？**
**A:** 不能。这是两个完全不同的平行宇宙。微信是围墙花园，DageChat 是荒野。它们在协议层、身份层和网络层都不兼容。DageChat 的目的不是连通微信，而是提供一个微信之外的避难所。

**Q8: 为什么启动画面里说“加载加密引擎”需要时间？**
**A:** 因为 DageChat 使用的是 **NIP-44 v2 (XChaCha20)** 和 **secp256k1** 等高强度加密算法，且完全在本地运行。为了确保安全性，软件启动时会初始化底层的 C 语言加密库（如 `nacl.bindings` 和 `coincurve`），并进行自检。这几秒钟的等待是安全计算的必要开销。

**Q9: 如果 Relay 是“哑”的，那它靠什么盈利？会倒闭吗？**
**A:** 许多 Relay 是由志愿者、加密社区或隐私组织运营的（类似 Tor 节点）。也有部分 Relay 采用付费订阅模式（NIP-42）。因为 Relay 成本很低（只存加密乱码，不管业务），所以即使某个 Relay 倒闭，用户只需在客户端配置里换一个 URL 即可，数据都在本地，不会丢失。

**Q10: “协议主权”对普通用户最大的好处是什么？**
**A:** **永不失联。** 在传统平台，如果你的号被封了，你不仅失去了说话的权利，还失去了所有好友联系方式。在 DageChat，因为身份（私钥）在你手里，好友关系（通讯录）在你本地硬盘里，没有任何力量能剥夺你的社交资产。你可以随时带着你的关系网迁移到任何地方。

## 第二章：系统架构与网络拓扑 (System Architecture)

> *"The network is the computer, but the client is the master."*
> — DageChat 架构哲学

### 2.1 客户端-中继器模型 (Client-Relay Model)：第三条道路

在构建抗审查网络时，通常有两种极端的选择：**中心化服务器（Client-Server）** 和 **纯 P2P（Peer-to-Peer）**。DageChat 选择了基于 Nostr 协议的 **Client-Relay** 模型，这是一条融合了两者优势的“第三条道路”。

**2.1.1 为什么不是纯 P2P (DHT/Gossip)？**
虽然 P2P（如 Tox, BitTorrent）在意识形态上很美，但在移动互联网时代存在严重工程缺陷：
*   **在线率问题**：如果 Alice 发消息给 Bob，而 Bob 手机断网或后台被杀，消息无法送达。P2P 要求双方同时在线。
*   **耗电与流量**：维护 DHT（分布式哈希表）网络需要移动设备不断进行“心跳”通信，这对手机电池是灾难性的。
*   **NAT 穿透**：大部分用户位于复杂的内网环境中，P2P 直连打洞成功率难以保证。

**2.1.2 为什么不是中心化 (Client-Server)？**
*   **单点故障**：服务器倒闭或被查封，全网瘫痪。
*   **数据垄断**：服务器拥有所有数据，用户被绑架。

**2.1.3 DageChat 的解法：存储与逻辑分离**
DageChat 的架构可以概括为：**“逻辑在端，存储在云（多点冗余）”**。
*   **中继器 (Relay)**：它不是服务器，它是**“云端死信箱”**。它像一个永远在线的邮箱，负责替不在线的用户暂存加密数据。它不处理任何业务逻辑。
*   **客户端 (Client)**：它是**“全功能节点”**。所有的身份验证、密钥协商、群组管理、黑名单过滤，全部由客户端在本地计算完成。

**2.1.4 多路复用与故障转移**
DageChat 客户端支持在 `relays.json` 中配置多个 Relay URL。
*   **写入策略**：客户端向所有连接的 Relay **广播**消息。只要有一个 Relay 存活，消息就能发布出去。
*   **读取策略**：客户端从所有 Relay **订阅**消息。本地数据库通过 Event ID 进行去重。这实现了天然的**高可用性 (HA)**。

**技术细节：独立管道架构**
源码 `client_persistent.py` 中的 `AsyncRelayManager` 为每个 Relay URL 分配了独立的 `connect_loop` 任务。这意味着 DageChat 的网络层是**非阻塞 (Non-blocking)** 的。即使你配置了一个位于地球另一端、延迟高达 5000ms 的 Relay，它也丝毫不会拖慢你与本地 Relay 的极速通信。

### 2.2 哑中继 (Dumb Relay)：基础设施的极简主义

DageChat 的服务端代码（`dagechat-relay-sqlite.py` / `dagechat-relay-redis.py`）刻意保持了极度精简。这种设计被称为 **"Dumb Relay, Smart Client"**。

**2.2.1 协议层的不可知论**
Relay 对数据内容表现为“不可知论”：
*   **看不懂内容**：`content` 字段是加密的 Base64 字符串（XChaCha20）。Relay 无法进行关键词审查或内容过滤。
*   **看不懂关系**：特别是对于 **幽灵群 (Type 1)**，Relay 看到的只是随机公钥向群公钥投递数据，无法识别这是否为群聊，更无法得知群成员列表。

**2.2.2 核心职责**
Relay 仅执行以下机械操作：
1.  **签名验证**：检查 `pubkey` 对应的 Schnorr 签名是否正确（防止伪造）。
2.  **PoW 校验**：检查 `id` 哈希的前导零是否满足 NIP-13 要求（防止垃圾轰炸）。
3.  **持久化**：将 JSON 存入 SQLite 或 Redis。
4.  **推送**：当有客户端发来 `REQ` 请求时，根据 Filter（过滤器）匹配并推送数据。

**2.2.3 性能优势**
因为剥离了所有业务逻辑（不需要查好友关系表、不需要鉴权、不需要解密），DageChat 的 Relay 性能极其强悍。一个基于 Redis 的单节点 Relay 理论上可支撑数万并发连接。

### 2.3 本地优先 (Local-First) 的数据策略

在 DageChat 中，云端数据是“暂时的”、“流动的”，而本地数据是“永恒的”、“主权的”。

**2.3.1 数据流向与并发控制**
*   **服务端**：`dagechat-relay-sqlite.py` 显式执行了 `PRAGMA journal_mode=WAL;`，利用 WAL 机制实现服务端的高吞吐并发。
*   **客户端**：`db.py` 使用 Python 标准库的 `threading.Lock` 进行同步控制。这种设计在单用户桌面环境下比 WAL 更稳健，避免了多进程争抢文件锁导致的数据库损坏风险，同时保证了 UI 线程与网络线程的数据同步。

**2.3.2 为什么本地存明文？**
源码 `db.py` 显示 `messages` 表存储的是解密后的内容。
*   **优势 1：极速检索**。本地全文搜索（Search Window）可以直接在 SQLite 中通过 `LIKE` 语句完成，无需遍历解密。
*   **优势 2：数据永存**。即使未来你的私钥丢失，或者群组更换了密钥，**只要你曾经解密过并存入了本地，这些历史记录就永远属于你**，可随时导出查看。这是“数据主权”的终极体现。
    *   *注：本地数据库文件本身依赖操作系统的文件权限保护，或者是全盘加密（BitLocker/FileVault）。DageChat 自身的加密重点在于网络传输和身份验证。*

**2.3.3 离线优先体验**
不同于 Web 版应用（如 Discord），断网时 DageChat 依然是一个功能完整的阅览器。你可以浏览历史、搜索内容、查看资料，这种体验接近于本地笔记软件。

### 2.4 软件目录结构与模块解析

通过解构源码目录，我们可以清晰看到架构的分层：

| 模块文件 | 架构层级 | 核心职责 |
| :--- | :--- | :--- |
| **`gui.py`** | **表现层 (UI)** | 程序入口。负责 CustomTkinter 界面渲染、用户交互事件响应、多线程调度。它是“皮囊”。 |
| **`client_persistent.py`** | **业务逻辑层 (Controller)** | 它是“大脑”。负责维护 WebSocket 长连接、心跳保活、消息分发路由 (`_handle_group_msg`, `_handle_dm`)、群组状态管理。 |
| **`nostr_crypto.py`** | **核心层 (Kernel)** | 它是“心脏”。封装了所有密码学原语（secp256k1, Schnorr, XChaCha20, HKDF）。这是系统安全边界的底线。 |
| **`db.py`** | **数据层 (Persistence)** | 负责 SQLite 的读写。实现了本地数据的持久化和查询优化（索引、批量插入）。 |
| **`gui_windows.py`** | **组件层 (Components)** | 独立的功能窗口（登录、设置、群管理、资料页）。实现了 UI 组件的模块化。 |
| **`lang_utils.py`** | **国际化层 (i18n)** | 负责多语言支持，实现了界面文案与逻辑代码的解耦。 |
| **`dagechat-relay-*.py`** | **基础设施层 (Infra)** | 服务端实现。独立于客户端运行，构成了去中心化网络的基础节点。 |

## 本章问答 (Q&A)

**Q1: 如果所有 Relay 都被关闭了，DageChat 还能用吗？**
**A:** 不能进行**新的**通信，但**历史**数据完全可用。
*   **短期影响**：消息发不出去。
*   **解决方案**：因为 Relay 软件也是开源的（且非常轻量），你或你的朋友可以在几分钟内搭建一个新的 Relay，并将地址分享给群友。大家把新地址填入客户端，通信立刻恢复。这种**“野火烧不尽”**的韧性是中心化软件无法比拟的。

**Q2: 为什么不使用现有的公共 Nostr Relay，而要自己开源 Relay 代码？**
**A:** 现有的公共 Nostr Relay（如 Damus 使用的）大多针对社交媒体（Twitter-like）优化，存储策略（TTL）较短，且通常开启了 NIP-42 鉴权或付费墙。DageChat 的 Relay 针对**即时通讯**进行了优化（支持高频写入、特定的 PoW 难度要求），且自建 Relay 能最大程度保护元数据隐私（不与无关人员共享连接）。当然，DageChat 客户端完全兼容标准的 Nostr Relay。

**Q3: “本地优先”意味着如果我换了电脑，聊天记录就没了吗？**
**A:** 是的，**默认情况下是这样**。
*   **原因**：为了安全，Relay 上的加密数据通常有过期时间（TTL，如30天），且 Relay 不会对历史数据负责。
*   **迁移方法**：你需要手动复制 `user_xxx` 文件夹到新电脑。
*   **哲学含义**：这就像纸质信件。信件在你抽屉里（本地硬盘），搬家时你需要自己带走它，而不是指望邮局（Relay）一直帮你存着。

**Q4: DageChat 的数据库没有加密吗？如果电脑被偷了怎么办？**
**A:** `dgchat.db`（或 `user.db`）中的消息内容是明文存储的，但**私钥**是经过高强度加密（Argon2 + SecretBox）存储的。
*   **如果电脑被偷**：黑客可以看到你的聊天记录（如果他能绕过操作系统密码直接读取硬盘），但他**无法导出你的私钥**（除非他有你的强密码），因此他无法以你的名义发消息。
*   **建议**：对于高安全需求用户，强烈建议开启操作系统的**全盘加密**（Windows BitLocker / macOS FileVault）。应用层的数据库加密在内存 dump 面前意义不大，依赖 OS 层加密更安全高效。

**Q5: 哑中继（Dumb Relay）能知道我在什么时候上线吗？**
**A:** 能。当你连接 WebSocket 时，Relay 知道你的 IP（如果不挂代理）和上线时间。
*   **防御**：这就是为什么我们强调使用 Tor。通过 Tor 连接 Relay，Relay 只能看到来自 Tor 出口节点的连接，无法关联到你的真实物理位置。

**Q6: 为什么不使用区块链来存储消息？那样不是更“永存”吗？**
**A:** 区块链（如 Ethereum, Solana）极其昂贵且缓慢（TPS 限制）。聊天是高频、低价值密度的行为（一句“在吗”不值得花 $5 Gas 费）。Nostr 协议巧妙地使用了**“区块链的公私钥技术”**来确权，但使用**“廉价的中继服务器”**来存数据。这是目前兼顾去中心化与性能的最优解。

**Q7: 多路复用（同时连多个 Relay）会消耗很多流量吗？**
**A:** 会增加一些，但对于文本聊天来说忽略不计。
*   **文本**：JSON 数据极小，即使广播给 5 个 Relay，流量也极低。
*   **图片/文件**：DageChat 目前传输图片是将 Base64 编码嵌入 JSON（这也是为了去中心化，不依赖特定的 OSS）。对于大图，确实会造成多倍流量消耗。未来版本可能会引入 NIP-94（文件附件分离存储）来优化这一点。

**Q8: 客户端启动时为什么要绑定那么多全局变量？**
**A:** 这是为了解决 PyInstaller 打包时的依赖丢失问题。在 Python 的动态特性和打包工具的静态分析之间，我们需要做一些“硬编码”的引导，确保打包后的 EXE 文件能在一个完全隔离的环境中找到所有需要的模块（如 `gui_windows`, `nostr_crypto`）。这属于工程实现的妥协。

**Q9: 我可以修改 Relay 代码来记录所有人的聊天吗？**
**A:** 你可以修改代码，但你**记录不到任何有意义的内容**。
*   **私聊/幽灵群**：你只能看到乱码，且不知道是谁发的（NIP-59）。
*   **标准群**：你能看到是谁发的，但内容依然是 XChaCha20 加密的乱码。
*   **结论**：恶意的 Relay 只能做“拒绝服务”（不给你存），做不到“窃听”（看不懂）。

**Q10: `client_persistent.py` 里的 `AsyncRelayManager` 是怎么工作的？**
**A:** 它是一个基于 `asyncio` 和 `aiohttp` 的异步管理器。
*   它维护一个 Event Loop。
*   为每一个 Relay URL 启动一个独立的 `Worker` 协程。
*   这些 Worker 并行工作，互不阻塞。即使连接了 10 个 Relay，其中 3 个超时，也不会影响 UI 的流畅度或其他 7 个 Relay 的收发。这是 DageChat 界面流畅的关键。

## 第三章：密码学基石 (Cryptographic Foundations)

> *"In code we trust."*
> — 密码朋克格言

DageChat 的安全模型建立在现代密码学原语之上，彻底摒弃了老旧的 NIST 标准（如 AES-CBC），转而采用更适合去中心化、高并发环境的 Bernstein 体系（Curve25519/ChaCha20）。v0.5.2 版本更是严格遵循并实施了 **NIP-44 v2** 标准，确立了以 **XChaCha20** 为核心的加密霸权。

### 3.1 身份体系：椭圆曲线的数字签名

在 DageChat 中，“身份”不是数据库里的一行记录，而是一个数学概念。

**3.1.1 曲线选择：secp256k1**
*   **选择原因**：这是比特币使用的椭圆曲线。相比 NIST 推荐的 P-256，`secp256k1` 的参数构造更透明，被普遍认为没有 NSA（美国国家安全局）植入的后门。
*   **公钥 (Public Key)**：32 字节的 X 坐标。这是你在网络上的公开地址（以 `npub1` 开头）。
*   **私钥 (Private Key)**：32 字节的随机数。这是你对身份的绝对控制权（以 `nsec1` 开头）。

**3.1.2 签名算法：Schnorr Signatures (BIP-340)**
*   **源码位置**：`nostr_crypto.py` -> `BIP340` 类及 `sign_event_id`。
*   **技术优势**：
    *   **线性特性 (Linearity)**：Schnorr 签名允许公钥和签名进行简单的数学加法。这为未来实现多重签名（Multi-Sig）和聚合签名（Threshold Signatures）奠定了基础（例如实现更安全的 DAO 治理群组）。
    *   **不可伪造性**：基于离散对数难题（DLP）。只要私钥不泄露，没有任何算力能伪造你的签名。
*   **性能优化**：代码优先调用 `coincurve` 库（libsecp256k1 的 C 语言绑定）进行签名和验证，速度是纯 Python 实现的数十倍，确保了移动端的流畅体验。只有在缺少 C 库环境时，才会回退到 `BIP340` 纯 Python 实现。

**容灾回退机制 (Fallback Resilience)**：
考虑到极端运行环境（如缺少 C 库支持的嵌入式系统），DageChat 内置了一套**纯 Python 实现的 BIP-340 算法栈**。当高性能 C 扩展加载失败时，系统会自动无缝切换至纯代码模式。这种“变色龙”式的适应能力确保了软件在任何 Python 能运行的地方都能保障核心签名功能的可用性。

### 3.2 通讯加密：XChaCha20-Poly1305 (NIP-44 v2)

这是 DageChat v0.5.2 所谓的“Secure Edition”的核心所在。

**3.2.1 为什么要抛弃 AES 和标准 ChaCha20？**
*   **AES-CBC 的缺陷**：存在填充预言机攻击（Padding Oracle Attacks）风险，且在没有硬件加速的设备上较慢。
*   **标准 ChaCha20 (IETF) 的致命伤**：它使用 **12 字节（96-bit）** 的 Nonce（随机数）。
    *   *生日悖论*：如果使用随机生成的 12 字节 Nonce，在发送大约 $2^{48}$ 条消息（或者更少，取决于安全阈值）后，发生 Nonce 碰撞的概率会变得不可忽视。
    *   *后果*：一旦两个不同的消息使用了相同的 Key 和 Nonce，攻击者可以通过异或操作直接还原出明文（Key Stream Reuse）。这对于长期存在的群聊是致命隐患。

**3.2.2 DageChat 的解法：XChaCha20**
*   **源码位置**：`nostr_crypto.py` -> `encrypt_nip44` / `encrypt_group_msg`。
*   **核心特性**：**24 字节（192-bit）Nonce**。
*   **安全性分析**：24 字节的随机空间大到不可思议。即使全人类每纳秒发送 10 亿条消息，直到宇宙热寂，发生 Nonce 碰撞的概率也无限趋近于零。
*   **实现细节**：
    *   代码调用 `nacl.bindings.crypto_aead_xchacha20poly1305_ietf_encrypt`。
    *   **Poly1305**：提供消息认证（MAC）。如果密文在传输过程中被中继器篡改哪怕 1 个比特，解密时会抛出异常，确保了**完整性**。

**3.2.3 密钥协商：ECDH + HKDF**
*   **ECDH**：利用椭圆曲线特性，Alice 用自己的私钥和 Bob 的公钥，可以计算出与 Bob 用私钥和 Alice 公钥计算出的**完全相同**的点（Shared Point）。
*   **HKDF-SHA256**：直接使用 ECDH 结果作为加密密钥是不安全的。代码使用 HKDF（基于 HMAC 的密钥派生函数）并混入 `salt=b'nip44-v2'`，将共享点派生为具备高熵值的 32 字节会话密钥。

### 3.3 路由混淆：NIP-59 Gift Wrap (俄罗斯套娃)

如果说 XChaCha20 保护了内容，那么 NIP-59 就保护了“谁在说话”。这是一种多层封装协议。

**3.3.1 第一层：谣言 (Rumor - Kind 14)**
*   **内容**：真实的聊天内容（JSON）。
*   **签名**：**发送者真实私钥**。
*   **目的**：向接收者证明“确实是我发的”。

**3.3.2 第二层：信封 (Seal - Kind 13)**
*   **操作**：将 Rumor 使用接收者的公钥进行 **NIP-44 加密**。
*   **签名**：**发送者真实私钥**。
*   **目的**：确保只有接收者能解开看到 Rumor。

**3.3.3 第三层：包装 (Wrap - Kind 1059)**
*   **核心魔法**：发送者在本地生成一个**临时的、一次性的随机密钥对** (Ephemeral Key)。
*   **操作**：使用这个**临时私钥**，将 Seal 加密给接收者。
*   **广播**：向 Relay 发送这个 Event。
*   **效果**：
    *   Relay 看到的发件人：`Random_Key_123`（一个从未见过的陌生人）。
    *   Relay 看到的收件人：`Receiver_Public_Key`（或者是群公钥）。
    *   **结论**：Relay 无法知道真实的发送者是谁。这就是**密封发送者 (Sealed Sender)** 技术。

### 3.4 协议版本控制与加密敏捷性

**3.4.1 协议头 (Version Byte)**
*   **源码体现**：`nostr_crypto.py` 中 `version = b'\x02'`。
*   **数据结构**：最终传输的 Base64 字符串解码后结构为：
    `[Version (1 byte)] + [Nonce (24 bytes)] + [Ciphertext (Variable)] + [MAC (16 bytes)]`
*   **战略意义**：
    *   `0x02` 代表当前使用的 XChaCha20。
    *   DageChat 客户端会检查首字节，若非 0x02 则拒绝处理（代码见 nostr_crypto.py）。这为未来引入抗量子算法预留了数据结构空间，届时只需更新客户端即可支持新算法，避免了协议层面的混乱。

### 3.5 安全指纹 (Safety Fingerprint)

**3.5.1 信任问题**
在去中心化网络中，当你第一次添加好友或进群时，你怎么知道对方公钥没有被中间人（MITM）替换？

**3.5.2 可视化验证**
*   **源码位置**：`client_persistent.py` -> `get_safety_fingerprint`。
*   **算法**：`SHA256(Key_Hex).hexdigest()[:6].upper()`。
*   **机制**：
    *   这是密钥的“数字指纹”。
    *   **雪崩效应**：如果黑客将公钥替换了哪怕 1 位，指纹就会面目全非。
    *   **带外验证 (OOB)**：用户通过电话、见面核对屏幕顶部的指纹。一致即安全。这是对抗 MITM 的终极防线。

## 本章问答 (Q&A)

**Q1: 为什么 DageChat 要自己实现 `NostrCrypto` 类，而不是直接调用库？**
**A:** DageChat 需要严格控制加密过程的每一个字节，以符合 NIP-44 v2 和 NIP-59 标准。例如，必须强制使用 24 字节 Nonce，必须在密文前添加 `0x02` 版本头。直接调用库无法满足这些特定的协议封装需求。封装类也是为了隔离底层库的变更，提供统一的接口。

**Q2: XChaCha20 和 AES-256 相比，哪个更安全？**
**A:** 在现代密码学观点中，两者在保密性上都被认为是“计算上不可破”的。但 XChaCha20 具有显著优势：
1.  **无硬件依赖**：在手机（ARM）上运行速度极快，且不容易受到缓存时序攻击（Cache-timing attacks），而 AES 往往依赖 CPU 指令集（AES-NI）才能安全高效。
2.  **Nonce 碰撞**：AES-GCM 通常使用 12 字节 Nonce，在大规模数据传输下有风险。XChaCha20 的 24 字节 Nonce 让它在“无状态”环境下具有压倒性的安全性。

**Q3: 如果 NIP-59 使用了临时密钥，接收者怎么知道是谁发的消息？**
**A:** 这就是“俄罗斯套娃”设计的精妙之处。Relay 只能解开最外层（Wrap），看到的是临时密钥。但接收者拥有私钥，可以剥开外层，露出中层（Seal）和内层（Rumor）。内层（Rumor）是由发送者的**真实私钥**签名的。所以，**只有接收者能确认发送者身份，其他人（包括 Relay）都不知道。**

**Q4: 安全指纹只有 6 位，会不会重复？**
**A:** 6 位 16 进制数有 $16^6 = 16,777,216$ 种组合。对于单个用户来说，遭遇随机碰撞的概率极低。但在对抗性攻击中，攻击者确实可能通过暴力计算生成一个“指纹相同但公钥不同”的伪造密钥（哈希碰撞攻击）。为了防御这种针对性攻击，在极高安全需求下，用户可以比对完整的公钥或更长的指纹。但这作为日常的便捷验证已经足够排除绝大多数机会主义攻击。

**Q5: 为什么加密函数里要用 `os.urandom(24)` 而不是 `random` 模块？**
**A:** `random` 模块是伪随机数生成器（PRNG），其种子通常基于时间，是可以被预测的。在密码学中，可预测等于可破解。`os.urandom` 调用操作系统的内核级熵源（如 `/dev/urandom`，基于硬件噪音、键盘敲击等），生成的是**密码学安全**的真随机数。

**Q6: DageChat 的群聊加密是“前向保密 (Forward Secrecy)”的吗？**
**A:** **严格来说，不是。** 目前使用的是静态共享密钥（群聊）或长期 ECDH 密钥（私聊）。如果你的私钥在一年后被盗，黑客可以解密你这一年来的所有流量（如果他存下来的话）。
**补救措施**：DageChat 依靠 Relay 的 **TTL（物理删除）** 策略来模拟前向保密。过期数据在物理上消失了，黑客即使拿到了密钥也因没有密文而无法解密。这是“存储侧的 PFS”。

**Q7: `secp256k1` 会被量子计算机破解吗？**
**A:** 会。Shor 算法理论上可以从公钥推导出私钥。这也是为什么我们在协议中预留了 **版本字节**。一旦后量子密码学（PQC）标准成熟（如 NIST 评选出的 Kyber），DageChat 可以通过升级协议版本无缝切换算法。

**Q8: 幽灵群的共享私钥机制，如果有一个成员叛变了怎么办？**
**A:** 这是一个**“一损俱损”**的模型。如果成员把私钥给了别人，那个人也就变成了合法的群成员。**但在网络层面上，叛变者无法证明“哪条消息是 Alice 发的”**。他只能泄露内容，无法泄露元数据（无法证明谁说了什么）。这就是幽灵群追求的“不可归因性”。

**Q9: 为什么代码中要引入 `cffi` 和 `_cffi_backend`？**
**A:** 因为 Python 的运算速度太慢了，无法支撑高强度的椭圆曲线运算。`coincurve` 和 `nacl` 都是 C 语言编写的高性能库，Python 通过 CFFI（C Foreign Function Interface）来调用它们。这也是为什么打包成 EXE 时需要特殊处理的原因。

**Q10: `HKDF` 中的 salt `nip44-v2` 是什么意思？**
**A:** 这是一个协议约定的常量盐值。它确保了所有遵循 NIP-44 v2 标准的客户端（不仅仅是 DageChat）都能派生出相同的会话密钥，从而实现互操作性。这是去中心化协议标准化的体现。

## 第四章：双模治理架构 (Dual Governance Architecture)

> *"Order is a feature for the many; Anarchy is a privilege for the few."*
> — DageChat 治理哲学

DageChat 并没有试图用一种通用的模型来解决所有沟通场景。相反，它承认人类需求的多样性，设计了两种截然不同的群组模式：**标准群组（Type 0）** 代表了基于共识的秩序，而 **幽灵群组（Type 1）** 则代表了数学上的绝对自由。

### 4.1 标准群组 (Type 0)：基于软性共识的秩序

这是我们在日常生活中最熟悉的群聊模式（类似微信群、Telegram群），但在去中心化架构下，其实现逻辑发生了根本性变化。

**4.1.1 “踢人”的物理失效**
在中心化服务器（如腾讯服务器）上，“踢出群聊”是一个物理动作：服务器不再转发消息给被踢者，也不再接收被踢者的消息。
但在 DageChat 的中继网络中，没有任何节点拥有“上帝权限”。
*   **无法阻断写入**：被踢者依然可以向 Relay 发送带有群ID标签的消息。Relay 作为一个“哑管道”，只要签名正确就会接收。
*   **无法阻断读取**：被踢者依然可以从 Relay 拉取群消息（虽然他没有密钥解不开，或者如果密钥未轮换，他依然能解开）。

**4.1.2 DageChat 的解法：软性共识治理**
既然无法物理切断连接，我们就通过**“全网广播 + 端侧过滤”**来实现逻辑上的屏蔽。
*   **权限锚定**：
    *   源码 `db.py` -> `groups` 表中存储了 `owner_pubkey`。
    *   群创建时，Owner 的公钥被写入群的创世区块（Metadata Event）。这是权力的唯一来源。
*   **指令广播**：
    *   源码 `client_persistent.py` -> `ban_group_member`。
    *   管理员构造一条特殊的 JSON 消息：`{"type": "ban", "pubkey": "Target_PK", ...}`。
    *   管理员使用自己的私钥对该指令进行签名，并发送到 Relay。
*   **端侧共识执行**：
    *   所有在线的群成员客户端接收到这条指令。
    *   **验证**：客户端检查签名是否属于该群的 `Owner`。
    *   **执行**：验证通过后，客户端将目标公钥写入本地 SQLite 的 `group_bans` 表。
    *   **渲染阻断**：当 UI 层渲染消息流时，会查询本地黑名单。如果发送者在黑名单中，该消息会被静默丢弃或折叠。

**4.1.3 哲学意义：代码契约**
这是一种民主与独裁的微妙平衡。
*   **表面独裁**：管理员依然拥有“一键屏蔽”的权力。
*   **实质民主**：这种权力不再是强制力的，而是基于**客户端代码的共识**。如果管理员滥用权力，社区成员可以修改开源客户端的代码（Fork），忽略该管理员的 Ban 指令，甚至分叉出一个新的群组。**治理权最终回归到了运行代码的用户手中。**

### 4.2 幽灵群组 (Type 1)：数学上的不可治理性

幽灵群是 DageChat 对“赛博朋克”精神的极致诠释。它不仅是为了隐私，更是为了创造一个**完全无法被审查、无法被问责**的言论空间。

**4.2.1 核心机制：共享私钥 (Shared Private Key)**
*   **传统模型**：Alice 用 Alice 的私钥签名。Bob 知道是 Alice 说的。
*   **幽灵模型**：群组 $G$ 生成一对密钥 $(PK_G, SK_G)$。
    *   入群不再是“把你拉进列表”，而是群主通过加密通道把 **群私钥 ($SK_G$)** 发给你。
    *   这就像把核发射井的钥匙复制了一份给你。

**4.2.2 身份擦除与路由封装**
源码 `client_persistent.py` -> `send_ghost_msg` 展示了发送过程：
1.  **内层签名**：使用群私钥 $SK_G$ 对消息进行签名。
    *   *意义*：证明了“我是群成员”（因为我有私钥）。
    *   *副作用*：抹除了“我是 Alice”的证据。
2.  **外层封装 (NIP-59)**：Alice 生成一个临时的随机密钥 $R$，将消息加密发送给群公钥 $PK_G$。
    *   *Relay 视角*：看到的是 `Random_Key(R) -> Group_PK(G)`。
    *   *流量特征*：这看起来就像是网络中无数个普通的点对点私聊中的一个。Relay 甚至不知道这是一个群聊消息，它以为这只是某人给 $G$ 发的一条私信。

**4.2.3 不可归因性 (Unattributability)**
这是幽灵群最可怕也最迷人的特性。
*   **对内匿名**：即使是群主，也无法知道哪条消息是哪个成员发的。大家共享同一个面具。
*   **对外隐形**：外部观察者无法统计群成员数量，无法构建社交图谱。
*   **不可治理**：
    *   **无法 Ban 人**：管理员无法针对性地屏蔽某个人，因为所有消息看起来都一样（都是群私钥签名的）。
    *   **一损俱损**：唯一的“治理”方式是废弃整个群（销毁 $PK_G$），但这会连累所有成员。

**4.2.4 幽灵的自我认知 (Self-Recognition Logic)**
在幽灵群中，所有成员共用一把私钥，这导致了一个逻辑悖论：客户端收到一条消息，解密后发现签名是自己手里的群私钥，那这条消息是“我”发的，还是“队友”发的？
*   **DageChat 的解法**：本地 `Nonce` 队列。
*   **机制**：发送时，客户端生成一个随机 Nonce 并存入内存队列 `my_ghost_nonces`。收到消息时，检查 payload 中的 Nonce 是否存在于队列中。
*   **效果**：实现了在“全员匿名”环境下的“自我识别”，且不依赖任何服务器状态。

### 4.3 密钥分发与隐形邀请 (Invisible Invitation)

在去中心化网络中，如何安全地将那个至关重要的“群私钥”或“对称密钥”交给新成员？

**4.3.1 隐形通道**
DageChat 利用 NIP-59 构建了一个隐蔽的邀请通道。
*   **封装**：邀请函（包含 ID, Key, Type）被打包成一个加密事件。
*   **投递**：通过 Gift Wrap 协议定向投递给受邀者。
*   **零元数据**：Relay 只能看到一串加密的 Blob，无法识别这是“邀请行为”，也无法知道邀请者是谁。

**4.3.2 信任的传递**
*   **普通群**：邀请意味着分享对称密钥。
*   **幽灵群**：邀请意味着**出让主权**（分享私钥）。
    *   源码 `gui.py` -> `open_ghost_invite_dialog` 中包含了一个高危弹窗警告。这不仅是 UI 设计，更是对用户责任的强调：**“你正在分享核按钮的钥匙，请确保对方值得信任。”**

## 本章问答 (Q&A)

**Q1: 在标准群中，如果我被管理员 Ban 了，我还能看到群里的消息吗？**
**A:** **理论上可以，但取决于密钥是否轮换。**
*   **现状**：目前的 v0.5.2 版本中，Ban 只是“客户端屏蔽”。如果你懂技术，直接向 Relay 请求数据并手动解密（如果你还持有旧的群密钥），你是能看到的。
*   **未来**：为了实现彻底的踢出，管理员需要执行 **Rekey（密钥轮换）**，即生成新的群密钥分发给未被 Ban 的成员。旧成员因为没有新密钥，就真的看不到了。

**Q2: 幽灵群里如果有人乱发广告或者捣乱怎么办？**
**A:** **毫无办法。**
这就是自由的代价。幽灵群的设计初衷是抗审查，而不是抗骚扰。一旦私钥分发出去，每个成员的权力都是平等的、不可剥夺的。如果群里混入了破坏者，唯一的办法是**解散群组**，核心成员带着新私钥去建立一个新群（Fork）。这迫使幽灵群只能保持小规模和高信任度。

**Q3: 为什么 Relay 无法封禁幽灵群？**
**A:** 因为 Relay 根本**识别不出**幽灵群。
在 Relay 看来，幽灵群的每一条消息都是“一个随机的陌生人给账号 G 发了一条私信”。这种流量特征与普通的 NIP-59 私聊完全一致。如果 Relay 试图封锁这种流量，它就必须封锁所有私聊功能，这将导致 Relay 失去服务能力。

**Q4: 我可以创建一个只有我自己的幽灵群吗？**
**A:** 可以。这实际上是一个极佳的**“云端加密笔记”**。你可以把重要信息发到这个群里，只有持有私钥的你（在任何设备上）能解密查看，而对于世界来说，这只是一堆指向虚空的乱码。

**Q5: 标准群的管理员权限可以转让吗？**
**A:** **目前不能。**
在 DageChat v0.5.2 中，群的所有权锚定在创建者的公钥上（硬编码在创世事件中）。这是为了简化信任模型。如果管理员丢失了私钥，该群将因失去管理能力而逐渐废弃（变成无政府状态）。

**Q6: “带外验证”在群邀请中如何体现？**
**A:** 当你收到一个群邀请时，你会看到群ID和群名。但黑客可能伪造一个同名的假群发给你。
**最佳实践**：在点击“加入”之前，通过电话问邀请人：“你发的那个群，ID的前几位是多少？”（虽然目前 UI 还没强制这一步，但在高安全需求下这是必须的）。

**Q7: 如果有人在幽灵群里泄露了群私钥，会发生什么？**
**A:** 灾难性的后果。
一旦群私钥公开，任何人都可以监听该群的历史和未来消息（失去保密性），任何人都可以冒充群成员发言（失去完整性）。这个群就实质上死亡了。这就是为什么我们称之为“一损俱损”。

**Q8: 为什么不使用 MLS (Messaging Layer Security) 协议来实现群聊？**
**A:** MLS 是 IETF 的新标准，确实支持更好的群组密钥管理（如树状结构更新）。但 MLS 极其复杂，且需要中心化服务器（Delivery Service）做排序的配合。DageChat 选择了更简单、更符合 Nostr 及其去中心化特性的方案（共享密钥/私钥），以换取架构的鲁棒性和无服务器特性。

**Q9: 幽灵群的成员上限是多少？**
**A:** 理论上没有上限。但从社会学角度看，**成员越多，信任越稀薄，私钥泄露的风险指数级上升**。建议幽灵群成员控制在 5-50 人以内的高信任圈子。标准群则可以支持数千人。

**Q10: “不可归因性”在法律层面意味着什么？**
**A:** 意味着**“无罪推定”**的技术实现。
即使执法机构拿到了一台存有幽灵群私钥的电脑，他们也无法在数学上证明“这条非法的消息是这台电脑的主人发送的”。因为群里其他 49 个人也拥有同样的私钥，都能生成同样的数字签名。这种“合理的推诿能力 (Plausible Deniability)”是保护异见者的最后屏障。

## 第五章：经济防御与反垃圾 (Economic Defense)

> *"Price is what you pay. Value is what you get. In a permissionless network, the price is paid in CPU cycles."*
> — DageChat 经济学

在无许可网络中，生成一个身份的成本几乎为零（只需运行一次 `secp256k1` 密钥生成算法）。这意味着攻击者可以瞬间生成数百万个虚假账号（女巫攻击），并向网络广播海量垃圾信息。传统的解决办法是引入验证码（CAPTCHA）或手机号验证，但这会破坏匿名性并引入中心化依赖。

DageChat 选择了 **NIP-13 (PoW)**：通过强制发送者消耗计算资源（算力/电费），来为网络写入操作定价。

### 5.1 分级工作量证明 (Tiered PoW) 原理

**5.1.1 核心机制：Hashcash**
DageChat 的 PoW 机制基于 Adam Back 发明的 Hashcash 算法（也是比特币的前身）。
*   **挑战**：在发送消息前，客户端必须找到一个随机数 `nonce`，使得整个消息事件 ID 的 SHA-256 哈希值满足特定的条件（即前 N 个比特位为 0）。
*   **公式**：
    $$ SHA256(Event\_Data + Nonce) < 2^{(256 - Difficulty)} $$
*   **非对称性**：
    *   **生成（挖矿）**：困难。客户端需要进行大量的暴力试错计算。
    *   **验证**：极易。Relay 只需进行一次哈希计算即可验证是否达标。

**5.1.2 源码解析**
*   **挖矿端 (`nostr_crypto.py`)**：
    `mine_pow_and_sign` 函数是一个死循环。它不断递增 `nonce` 标签的值，重新序列化 JSON，计算 Hash，直到找到满足 `difficulty` 的哈希值为止。
    *   *代码细节*：为了防止界面假死（GUI Freeze），虽然核心计算在 C 扩展中很快，但在 Python 循环层，对于极高难度可能需要多线程处理（客户端采用了 `threading` 后台挖矿）。
*   **验证端 (`dagechat-relay-*.py`)**：
    `verify_pow` 函数极其简单。它将 Event ID (Hex) 转为整数，直接与目标阈值比较。如果不满足，Relay 会直接返回 `["OK", eid, False, "pow: difficulty too low"]` 并断开连接。

### 5.2 难度分级详解 (Tiered Strategy)

DageChat 并没有“一刀切”地应用 PoW，而是根据**信任场景**实施了分级策略（定义在 `client_persistent.py` 中）。

#### 5.2.1 Level 1: 官方大厅 (Difficulty 16)
*   **场景**：`OFFICIAL_GROUP_ID`。这是一个全网公开、任何人只需知道 ID 就能进入的场所。它是垃圾广告、脚本机器人的首要攻击目标。
*   **参数**：`OFFICIAL_GROUP_POW_DIFFICULTY = 16`。
*   **成本分析**：
    *   要求哈希值前 16 位为 0。
    *   期望尝试次数：$2^{16} = 65,536$ 次哈希计算。
    *   **时间成本**：在普通 PC 上约需 0.5 ~ 2 秒；在手机上可能需 3~5 秒。
*   **战略目的**：**经济阻断**。发送一条广告需要消耗数秒的 CPU 满载时间。这对于正常聊天的用户是可以忍受的（发一句话等一会），但对于试图每秒发送 1000 条广告的机器人来说，算力成本将变得不可承受。

#### 5.2.2 Level 2: 普通群组 (Difficulty 8)
*   **场景**：用户自建的群组。通常通过邀请码加入，有一定的信任基础。
*   **参数**：`NORMAL_GROUP_POW_DIFFICULTY = 8`。
*   **成本分析**：
    *   期望尝试次数：$2^8 = 256$ 次。
    *   **时间成本**：毫秒级（几乎无感）。
*   **战略目的**：**技术防抖**。这主要不是为了防黑客，而是为了防止客户端 bug（如死循环发送）或低级脚本造成的意外洪泛。它是一个基本的门槛，表明“这是一个合法的客户端行为”。

#### 5.2.3 Level 3: 幽灵群与私聊 (Difficulty 0)
*   **场景**：高信任域。幽灵群需要私钥邀请，私聊通常基于好友关系。
*   **参数**：`0`。
*   **成本分析**：无需挖矿。
*   **战略目的**：**性能优化**。
    *   幽灵群和私聊使用的是 **NIP-59** 协议。
    *   NIP-59 的封装过程极其复杂：需要生成临时密钥对（ECDH）、加密 Seal（XChaCha20）、再次加密 Wrap（XChaCha20）。
    *   **天然的 PoW**：非对称加密本身就是一种计算密集型操作（Cost-of-Cryptography）。这构成了天然的算力壁垒。因此，无需再额外增加 PoW 负担，以节省移动端电量。

## 本章问答 (Q&A)

**Q1: 为什么我在官方大厅发消息的时候，界面会卡顿一下或者转圈？**
**A:** 这是正常的。因为你的电脑正在进行 **Difficulty 16** 的 PoW 挖矿。你的 CPU 正在疯狂计算哈希值，以向网络证明你不是垃圾机器人。这种短暂的延迟是你为维护社区清洁所支付的“过路费”。

**Q2: 手机算力比较弱，会不会永远挖不出 Difficulty 16？**
**A:** 不会永远挖不出，但会很慢，且非常耗电。这是去中心化反垃圾的代价。如果手机端体验太差，用户可以建议管理员降低难度，或者只在桌面端参与高频公共讨论。对于私聊（Difficulty 0），手机端没有任何压力。

**Q3: 黑客可以用 ASIC 矿机（比如比特币矿机）来攻击 DageChat 吗？**
**A:** 理论上可以，但**不划算**。
1.  **协议不兼容**：比特币矿机是针对双重 SHA-256 block header 优化的，而 Nostr Event 的 JSON 序列化格式不同，无法直接使用现成矿机。
2.  **经济账**：即使黑客定制了 FPGA/ASIC，他投入巨资只是为了在一个聊天软件里发垃圾广告？这种攻击的投入产出比（ROI）极低。PoW 的目的不是绝对防御，而是**拉高攻击成本**。

**Q4: 为什么不用验证码（Captcha）？**
**A:** 验证码需要一个中心化的服务器来生成和验证（如 Google reCAPTCHA）。这违背了 DageChat 的去中心化原则。如果不小心引入了 Google 的验证码，Google 就能知道谁在什么时候用了 DageChat，这是不可接受的隐私泄露。PoW 是唯一纯数学、去中心化的反垃圾方案。

**Q5: 如果有人用超级计算机在官方大厅刷屏怎么办？**
**A:** 这时 **“主权治理”**（第三章）就派上用场了。
PoW 是第一道防线（经济防御）。如果被突破，管理员可以发布签名的 `ban` 指令。此时，黑客付出了昂贵的算力成本发送垃圾信息，结果所有人的客户端都在本地将其屏蔽了。黑客亏损了电费，且没有达到骚扰效果。

**Q6: 我自建的 Relay 可以调高难度吗？比如要求 Difficulty 20？**
**A:** 当然可以。这是你的 Relay 主权。
在 `dagechat-relay-*.json` 配置文件中，并没有硬编码难度，但在代码逻辑中你可以修改 `verify_pow` 的阈值。如果你把难度调到 20，那么只有愿意为你付出几十秒算力的人才能把消息存入你的 Relay。这可以打造一个极高质量的精英社区。

**Q7: PoW 会增加流量消耗吗？**
**A:** 不会。PoW 消耗的是 **CPU/电力**，不消耗网络带宽。实际上，它通过遏制垃圾信息，反而**节省**了全网的带宽和存储空间。

**Q8: 幽灵群为什么不设 PoW？如果有人在幽灵群里刷屏怎么办？**
**A:**
1.  **技术原因**：幽灵群的消息结构（NIP-59）使得 Relay 无法验证内部内容的 PoW（因为 Relay 解不开包）。如果要把 PoW 加在外层（Wrap），会增加额外的计算负担。
2.  **社会原因**：幽灵群是基于私钥分享的高信任群组。如果你把私钥给了一个会刷屏的人，这属于**信任崩塌**，不是技术问题。正如你把家门钥匙给了疯子，门锁再好也没用。

**Q9: 这里的 PoW 和比特币的 PoW 有什么区别？**
**A:** 原理相同（Hashcash），但**目的不同**。
*   **比特币**：为了全网共识，争夺记账权，产生新货币。是竞争性的。
*   **DageChat**：为了反垃圾，增加发送成本。是非竞争性的（你挖你的，我挖我的，互不影响）。

**Q10: 未来会支持“付费免 PoW”吗？**
**A:** 这是 Relay 运营者的选择。
NIP-42 协议支持 Relay 进行身份验证。有些 Relay 可能会向付费白名单用户提供 `Difficulty 0` 的写入权限，而对免费用户要求 `Difficulty 20`。DageChat 客户端未来可以支持这种逻辑，这将形成一个可持续的 Relay 经济生态。

## 第六章：威胁模型与隐私全景 (Threat Model & Privacy)

> *"Privacy is not about hiding something; it's about defining who has the power to see."*
> — DageChat 安全哲学

在使用任何安全软件之前，必须明确：**你的对手是谁？** DageChat 的架构设计针对了三种特定类型的敌手模型，并提供了不同等级的防御策略。

### 6.1 敌手名录 (The Adversary Roster)

**6.1.1 敌手 A：恶意中继器 (The Malicious Relay)**
*   **能力**：拥有数据库的完整读写权限。可以记录所有经过该节点的 Event 数据。
*   **动机**：商业数据挖掘、配合执法取证、或仅仅是管理员的好奇心。
*   **DageChat 的防御**：
    *   **内容盲区**：所有内容经 XChaCha20 加密，Relay 只能看到乱码。
    *   **关系盲区**：对于私聊和幽灵群，使用 NIP-59 封装，Relay 甚至不知道谁是发送者。

**6.1.2 敌手 B：全局被动监控者 (Global Passive Adversary - GPA)**
*   **能力**：ISP（互联网服务提供商）、防火墙或国家级监控机构。他们能看到全网的流量包，进行流量分析（Traffic Analysis）。
*   **动机**：构建社会关系图谱（Social Graph），识别异见者网络。
*   **DageChat 的防御**：
    *   **传输层加密**：WebSocket over TLS (wss://) 防止了简单的嗅探。
    *   **不可归因性**：幽灵群的流量特征与普通私聊完全一致，在宏观流量中无法被特征识别。

**6.1.3 敌手 C：主动中间人 (Active MITM Attacker)**
*   **能力**：位于用户与 Relay 之间（如公共 WiFi），能够拦截、篡改、伪造数据包。
*   **动机**：在密钥交换阶段替换公钥，从而劫持会话。
*   **DageChat 的防御**：
    *   **安全指纹 (Safety Fingerprint)**：强制的人工带外验证。

**6.1.4 邀请码的安全校验**
DageChat 的邀请码不仅仅是密钥的拼接，还包含了一个**哈希校验位 (Checksum)**。
*   **算法**：`SHA256(GID + Key + Type + Salt)[0:6]`。
*   **目的**：防止用户在复制粘贴时遗漏字符，或防御恶意攻击者将“普通群”伪装成“幽灵群”诱骗用户导入。任何对邀请码的微小篡改都会导致校验失败，客户端将拒绝执行加入操作。

### 6.2 隐私泄露矩阵 (Privacy Leakage Matrix)

这是 DageChat 各模式下隐私保护能力的真实写照。

| 数据维度 | **私聊 (DM)** | **标准群 (Standard)** | **幽灵群 (Ghost)** | **传统 IM (微信/TG)** |
| :--- | :--- | :--- | :--- | :--- |
| **内容 (Content)** | **加密** (XChaCha20) | **加密** (XChaCha20) | **加密** (XChaCha20) | 仅传输加密 / 服务端可见 |
| **发送者身份** | **隐匿** (Sealed) | **暴露** (需签名验证) | **隐匿** (共享私钥) | 暴露 (实名) |
| **接收者身份** | **隐匿** (Wrapped) | 暴露 (群ID可见) | **隐匿** (Wrapped) | 暴露 |
| **社交图谱** | **Relay 不可知** | Relay 可构建 | **Relay 不可知** | 平台完全掌握 |
| **IP 地址** | 暴露给 Relay* | 暴露给 Relay* | 暴露给 Relay* | 暴露给平台 |

*注：IP 地址暴露是 TCP/IP 协议的物理特性。DageChat 强烈建议配合 **Tor (洋葱路由)** 使用，以实现 IP 层的匿名。*

### 6.3 深度解析：为什么标准群要暴露身份？

在 `6.2` 矩阵中，你会发现 **标准群 (Type 0)** 是唯一暴露发送者身份和群 ID 的模式。这是设计上的权衡。

*   **为了治理**：在第三章提到，标准群通过“软性共识”来实现 Ban（屏蔽）功能。为了让客户端能执行 `ban` 指令，客户端必须知道“这条消息是谁发的”。如果发送者匿名（如幽灵群），就无法实现针对性的屏蔽。
*   **为了效率**：标准群使用对称密钥广播。Relay 可以根据 `kind: 42` 和 `tags: [['g', group_id]]` 高效地索引和推送消息。
*   **结论**：标准群适用于**公开/半公开社区**（如技术讨论组），这里的需求是“秩序 > 极致隐私”。

### 6.4 深度解析：幽灵群的“黑洞”特性

**幽灵群 (Type 1)** 是 DageChat 对抗“全局监控者”的终极武器。

*   **流量伪装**：在网络层面，幽灵群的通信看起来就像是：`随机公钥 A -> 随机公钥 B`。
*   **不可区分性**：
    *   如果你和朋友私聊，流量是 `Random -> Target`。
    *   如果你在幽灵群发言，流量是 `Random -> Group_PK`。
    *   对于监控者来说，幽灵群的流量特征完美伪装成了**“粉丝向大V发送私信”**（Many-to-One）。这为用户提供了极强的**合理的推诿性 (Plausible Deniability)**：即使被观测到向特定地址发送数据，用户也可以主张自己仅是在向某位公众人物发送加密私信，从而在宏观流量分析中隐藏了“群组协同”的意图。
*   **零图谱**：因为无法区分，监控者就无法画出“谁在群里”的关系图。群成员就像潜伏在黑夜中的幽灵，聚散无形。

### 6.5 安全指纹：反中间人攻击的最后防线

**6.5.1 信任的锚点**
在去中心化网络中，没有 CA（证书颁发机构）告诉你“这个公钥确实属于 Alice”。如果黑客 Eve 在中间拦截，给了你 Eve 的公钥，你加密发出的消息就会被 Eve 解密。

**6.5.2 代码实现**
源码 `client_persistent.py` 中的 `get_safety_fingerprint` 函数计算并显示了一个哈希摘要：
$$ Fingerprint = \text{SHA256}(Target\_Key)[0:6] $$

**6.5.3 验证仪式 (The Ceremony)**
DageChat 要求用户执行一种“仪式”：
1.  Alice 在 DageChat 上添加 Bob。
2.  Alice 看到指纹 `C4F2A1`。
3.  Alice 打电话给 Bob（或当面）问：“你的指纹是多少？”
4.  Bob 回答：“`C4F2A1`”。
5.  **验证通过**。

如果 Bob 回答是 `998877`，说明网络中存在**中间人攻击**，或者 Alice 加错人了。**立即停止通信！**

### 6.6 前向保密 (Forward Secrecy) 的现实

现代 IM（如 Signal）使用双棘轮算法实现前向保密（PFS）：即使密钥泄露，也只能解密一条消息，不能解密历史记录。
DageChat v0.5.2 **尚未** 实现复杂的双棘轮（为了保持架构的简洁和离线可用性）。

**DageChat 的替代方案：物理删除 (TTL)**
*   **机制**：Relay 设置了数据保留期限（如 30 天）。
*   **效果**：
    *   假设黑客在第 31 天偷到了你的私钥。
    *   他去 Relay 下载数据，发现第 1-30 天的数据已经被**物理擦除**了。
    *   他无法解密历史，因为历史已经不存在于网络上了。
*   **结论**：这是基于存储侧的“延迟前向保密”。虽然不如双棘轮完美，但在工程上极其有效且鲁棒。

## 本章问答 (Q&A)

**Q1: 使用 DageChat 会被运营商（ISP）发现吗？**
**A:** ISP 可以看到你在使用 DageChat 协议（如果未使用 Tor），因为 WebSocket 流量特征是可见的。ISP 知道你连接了哪个 Relay，消耗了多少流量。
**但是**，ISP **绝对无法知道**你在说什么，你也无法知道你在和谁说（除非你们都在标准群且 ISP 也在监听该群）。内容和关系图谱是安全的。

**Q2: 如果我不用 Tor，Relay 能看到我的 IP 吗？**
**A:** **能。** 这是 TCP/IP 协议决定的。当你连接 Relay 时，必须建立 TCP 连接，Relay 自然会获得你的源 IP。
**建议**：如果你处于极高风险环境，**必须**在 `relays.json` 中配置 `proxy` 字段指向本地的 Tor 端口（如 `127.0.0.1:9050`）。

**Q3: 幽灵群的安全性这么高，为什么不把它设为默认？**
**A:** 因为它牺牲了便利性和治理能力。
1.  **性能**：NIP-59 封装需要三次非对称加密，比标准群（一次对称加密）慢且耗电。
2.  **治理**：幽灵群无法踢人。如果进了一个发广告的，除了炸群重组别无他法。
3.  **密钥风险**：幽灵群共享私钥。一旦私钥泄露，整个群的历史记录对泄露者可见。
因此，幽灵群被定位为“特种作战工具”，而非日常工具。

**Q4: 安全指纹为什么只有 6 位？够用吗？**
**A:** 6 位 16 进制数有 1600 万种组合。
对于防止“随机错误”或“盲目攻击”，足够了。
但如果是一个拥有无限算力的国家级黑客，确实可能通过暴力计算生成一个“指纹碰撞”的假公钥。
**进阶防御**：如果你面临这种级别的威胁，请在代码中修改截取长度，比对完整的 64 位公钥。对于 99.9% 的用户，6 位是一个兼顾体验（易于口述）和安全的最优解。

**Q5: 如果中继器保存了我的加密消息，他们能暴力破解 XChaCha20 吗？**
**A:** 不能。XChaCha20 使用 256 位密钥。以人类目前的物理学认知和算力总和，暴力穷举 256 位密钥所需的能量超过了将地球海洋烧开所需的能量。除非量子计算机取得突破性进展（即使那样，也需要数十年）。

**Q6: DageChat 能防止“截屏”或“拍照”吗？**
**A:** **不能。** 这属于“模拟漏洞 (Analog Hole)”。没有任何软件能阻止物理世界的相机拍摄屏幕。
**防御**：不要在任何电子设备上发送会让你进监狱的信息，除非你信任对方的物理环境安全。

**Q7: 为什么说 DageChat 的威胁模型是“反脆弱”的？**
**A:** 塔勒布定义的“反脆弱”是指从冲击中受益。
*   如果一个中心化服务被攻击，它会瘫痪（脆弱）。
*   如果 DageChat 的一个 Relay 被封锁或攻击，用户会迁移到其他 Relay，或者搭建新的 Relay。攻击行为反而会促使网络变得更加去中心化、节点更多、生命力更强。

**Q8: 我可以信任开源的 Relay 代码吗？会不会有后门？**
**A:** 永远不要信任，要验证（Don't trust, verify）。
源代码是公开的 (`dagechat-relay-*.py`)，代码量非常少（几百行）。任何懂 Python 的人都可以审计它。你会发现它只是简单地把接收到的 JSON 存进数据库，没有任何复杂的逻辑，更没有上传数据的“后门”。

**Q9: 如果我的私钥被盗了，但我要怎么告诉朋友不要信这个号了？**
**A:** 这是一个难题。在去中心化网络中，拥有私钥就等于拥有身份。你无法“注销”被盗的账号。
**补救**：你需要通过带外渠道（电话、见面）告诉朋友：“我的旧号被盗了，这是我的新号公钥，请验证新号的安全指纹。” 然后所有人拉黑旧号。

**Q10: DageChat 怎么处理元数据中的“时间戳”泄露？**
**A:** 这是一个高级话题。即使内容加密，消息的**发送时间**也是公开的。通过分析时间模式（Timing Analysis），攻击者可能推测出“Alice 和 Bob 经常在同一时间段活跃”。
**当前状态**：DageChat v0.5.2 尚未混淆时间戳。
**未来展望**：未来的版本可能会引入“延迟发送”或“填充流量 (Padding traffic)”机制，人为制造网络噪音来掩盖真实的时间模式。

## 第七章：数据库与本地存储 (Database Schema & Persistence)

> *"Your keys, your data. Not your keys, not your data."*
> — 区块链与 DageChat 的共同信条

DageChat 采用 **“本地优先 (Local-First)”** 架构。这意味着客户端不仅仅是一个显示终端（像 Web 网页那样），而是一个**全功能的轻节点**。它拥有完整的历史数据副本，具备独立的索引、搜索和解密能力。

### 7.1 SQLite 表结构深度解析 (Schema Anatomy)

DageChat 使用 **SQLite3** 作为存储引擎。这是一个无服务器的、零配置的、事务性的 SQL 数据库引擎。根据 `db.py` 的源码，我们定义了以下核心表结构：

#### 7.1.1 核心资产：`account` 表
这是整个数据库中最敏感的表，存储了用户的身份凭证。
*   **`pubkey` (TEXT)**: 用户的公钥（Hex 格式）。这是公开身份。
*   **`crypto_blob` (TEXT)**: **加密后的私钥数据包**。
    *   *结构*：这是一个 JSON 字符串，包含 `{"salt": "...", "nonce": "...", "ciphertext": "..."}`。
    *   *安全机制*：私钥**绝不会**以明文形式存储。即使你打开了 `.db` 文件，看到的也只是一串乱码。必须输入启动密码才能解开这个 Blob。

#### 7.1.2 社交关系：`contacts` 表
存储用户的通讯录和好友关系。
*   **`pubkey`**: 好友的身份公钥。
*   **`enc_key`**: **加密公钥**。在 NIP-44 协议中，我们需要对方的公钥来进行 ECDH 协商。通常 `enc_key` 等于 `pubkey`，但在某些复杂的密钥轮换场景下可能不同。
*   **`is_blocked`**: 本地屏蔽标记。如果设为 `1`，来自该公钥的消息将在底层被丢弃，不会渲染到 UI。

#### 7.1.3 群组凭证：`groups` 表
存储群组的元数据和密钥。
*   **`group_id`**: 群的唯一标识（对于幽灵群是公钥，对于普通群是哈希 ID）。
*   **`key_hex`**: **核心机密**。
    *   对于 **标准群 (Type 0)**：这是共享的 **对称密钥 (Symmetric Key)**。
    *   对于 **幽灵群 (Type 1)**：这是共享的 **群私钥 (Shared Private Key)**。
*   **`type`**: 决定了客户端使用哪种加密逻辑（`encrypt_group_msg` vs `make_gift_wrap`）。

#### 7.1.4 历史记忆：`messages` 表
这是体积最大的一张表，存储了所有的聊天记录。
*   **`content`**: **解密后的明文 (Plaintext)**。
    *   *设计哲学*：为什么存明文？为了**数据主权**和**检索性能**。
    *   如果存密文，每次查看历史记录都需要重新解密，消耗巨大 CPU；且如果未来密钥丢失，历史记录将永久锁死。存明文确保了“一次解密，永久持有”。
*   **`sender_pubkey`**: 发送者。
    *   *幽灵群特性*：在幽灵群中，这个字段存储的是群公钥（因为无法识别个人），但在内容 JSON 的 `alias` 字段中可能包含发送者自称的昵称。
*   **`group_id`**: 消息归属。

#### 7.1.5 治理工具：`group_bans` 表
这是“软性共识治理”的本地实现。
*   **`group_id` & `banned_pubkey`**: 定义了“在哪个群屏蔽谁”。
*   **工作流**：当客户端收到管理员签名的 `ban` 指令时，会自动向此表插入记录。渲染引擎在显示消息前，会先查询此表（`is_banned_in_group`），实现过滤。

### 7.2 本地私钥保护机制 (Private Key Security)

源码 `client_persistent.py` 中的 `_encrypt_and_save` 和 `unlock_account` 展示了极高规格的本地安全防护。

**7.2.1 密钥派生：Argon2**
当用户设置启动密码时，系统并不是直接用密码加密私钥，而是使用了 **Argon2i** 变体算法。
*   **抗侧信道攻击 (Side-Channel Resistance)**：`Argon2i` 采用与数据无关的内存访问模式。即使攻击者在物理机上监控 CPU 缓存时序（Cache Timing），也无法通过执行时间差异反推密码特征。这是针对**“被入侵设备”**场景的最后一道防线。
*   **内存硬依赖**：消耗大量 RAM，有效抵御 GPU 集群暴力破解。

**7.2.2 对称加密：SecretBox (XSalsa20)**
使用 Argon2 派生出的密钥，结合 `nacl.secret.SecretBox` 对私钥进行加密。这保证了即使数据库文件被盗，攻击者在没有密码的情况下，数学上无法还原出私钥。

**7.2.3 内存安全**
私钥仅在软件运行时解密并驻留在内存（RAM）中。软件关闭后，RAM 释放，私钥消失。DageChat 不会在任何临时文件或日志中写入明文私钥。

**7.2.4 静态数据安全边界 (Critical)**
必须明确的是，DageChat 的本地加密重点在于保护**用户身份私钥 (Identity Private Key)**。
*   **受保护**：您的身份私钥（`nsec`）经 Argon2 + SecretBox 加密存储。
*   **未加密**：您的**群组共享密钥**及**聊天记录明文**直接存储于 SQLite 文件中。
*   **安全建议**：如果攻击者窃取了您的数据库文件，虽然无法冒充您发言（没私钥），但可以查看历史聊天。因此，**强烈建议启用操作系统级的全盘加密（如 Windows BitLocker / macOS FileVault）**作为最后一道防线。

### 7.3 文件附件的处理

目前 DageChat 采用**内嵌式存储**。
*   **机制**：图片被转换为 Base64 编码，直接嵌入在 JSON 消息体中，并存入 SQLite 的 `content` 字段。
*   **优点**：**极致的去中心化**。图片不依赖任何第三方图床或 OSS（对象存储服务）。只要你能连上 Relay，图片就能送达；只要你有数据库，图片就永远存在。
*   **代价**：数据库体积膨胀较快。

## 本章问答 (Q&A)

**Q1: 为什么不在本地数据库里也存密文，每次看的时候再解密？这样不是更安全吗？**
**A:** 这样做有严重的可用性问题。
1.  **搜索失效**：如果存密文，你就无法使用“全局搜索”功能瞬间找到半年前某句话。
2.  **密钥依赖**：如果你修改了群密钥（Rekey），旧的密文可能就永远打不开了。
3.  **性能**：每次滚动聊天记录都要实时解密几十条消息，手机会发烫掉电。
**安全建议**：依赖操作系统的全盘加密（BitLocker/FileVault）来保护本地文件，而不是在应用层做低效的重复加密。

**Q2: 数据库文件 `dgchat.db` 能直接复制到另一台电脑用吗？**
**A:** **完全可以。**
这正是 DageChat 的优势。你可以把这个文件复制到 U 盘，插到另一台电脑上，安装 DageChat，输入同样的密码（因为私钥加密依赖密码），瞬间恢复所有聊天记录、好友和群组。这是真正的**“数据携带权”**。

**Q3: 随着聊天记录增多，SQLite 数据库会卡顿吗？**
**A:** SQLite 非常强大，支持 TB 级别的数据。`db.py` 中已经为 `created_at`、`group_id` 等字段建立了**索引 (INDEX)**。即使有几十万条消息，查询速度也是毫秒级的。

**Q4: 如果我忘记了启动密码，能通过数据库文件找回私钥吗？**
**A:** **绝对不能。**
Argon2 算法就是为了防止这种情况设计的。没有密码，数学上无法解开 `crypto_blob`。你的账号将永久丢失。这也提醒用户：**务必离线备份私钥（nsec 字符串）**，而不仅仅是依赖数据库文件。

**Q5: 为什么图片要转成 Base64 存数据库，而不存文件路径？**
**A:** 为了**原子性（Atomicity）**。
如果存文件路径，当你迁移数据（复制 db 文件）时，很容易忘记复制图片文件夹，导致聊天记录里的图全都裂了。将图片内嵌在数据库中，确保了“一个文件包含所有”，极大地简化了数据备份和迁移的难度。

**Q6: 怎么清理数据库空间？**
**A:** DageChat 提供了“清空聊天记录”的功能。
在代码底层，这执行的是 `DELETE FROM messages`。为了释放磁盘空间，你可能需要手动执行 SQLite 的 `VACUUM` 命令（或者未来版本会自动处理）。

**Q7: 本地数据库会被病毒扫描或上传吗？**
**A:** 取决于你的操作系统安全环境。
DageChat 只是一个普通软件，它无法阻止拥有管理员权限的病毒读取你的文件。因此，**不要在不安全的电脑上运行 DageChat**。对于极高危环境，建议使用 Tails OS 或虚拟机。

**Q8: `group_bans` 表是永久的吗？**
**A:** 是的，直到管理员发送 `unban` 指令。
有趣的是，因为它是本地存储，用户作为数据库的主人，其实可以**手动修改数据库**（比如用 SQLite 浏览器打开），删掉黑名单里的记录，从而单方面“解除屏蔽”。这进一步体现了用户对客户端的绝对控制权。

**Q9: 幽灵群的私钥也是加密存储在 `groups` 表里的吗？**
**A:** 不，群组的 `key_hex` 是**明文**存储在 `groups` 表里的。
因为群聊需要频繁自动接收消息，如果群密钥也加密，后台接收消息时就需要用户不断输入密码，体验极差。这再次强调了**操作系统层全盘加密**的重要性。

**Q10: 如果 Relay 发送了恶意的 SQL 注入代码怎么办？**
**A:** DageChat 使用了 SQLite 的**参数化查询 (`?` 占位符)**。
在 `db.py` 的所有 SQL 语句中，变量都是通过 `(?, ?, ?)` 传入的，而不是字符串拼接。这在底层彻底杜绝了 SQL 注入的可能性。Relay 发送的任何怪异字符都只会被当作普通文本处理。

**Q11: 为什么源码 `gui.py` 头部有一段永远不会执行的 `if False: import ...` 代码？**
**A:** 这是一个针对 PyInstaller 打包工具的工程化处理（Hook）。由于 DageChat 使用了动态模块加载和复杂的 C 语言加密库绑定，静态分析工具往往无法识别这些依赖。这段“死代码”强制打包工具将所有必要的加密库和子窗口模块包含在最终的 EXE 文件中，确保了程序的独立运行能力。

## 第八章：中继器服务端实现 (Relay Implementation)

> *"Be a pipe, not a bucket. Flow, do not hold."*
> — DageChat 中继器设计哲学

DageChat 的服务端不叫“Server”，而叫“Relay”（中继器）。这两个词的区别在于权力和责任：Server 意味着管理与控制，Relay 意味着传递与服务。为了适应不同的部署场景，DageChat 官方开源了两个版本的实现：**轻量级 SQLite 版** 和 **高性能 Redis 版**。

### 8.1 双版本策略：去中心化的基础设施

**8.1.1 为什么要搞两个版本？**
去中心化不仅意味着没有中心节点，还意味着**节点的异构性（Heterogeneity）**。
*   **SQLite 版 (`dagechat-relay-sqlite.py`)**：为了**“个人主权”**。
    *   *目标*：让不懂技术的用户也能在几分钟内，在一台破旧的笔记本、树莓派甚至安卓手机（Termux）上跑起一个节点。
    *   *特点*：单文件，零依赖（除了 Python 库），数据就是一个 `.db` 文件，随拷随走。
*   **Redis 版 (`dagechat-relay-redis.py`)**：为了**“公共服务”**。
    *   *目标*：为成千上万的用户提供高并发、低延迟的消息路由服务。
    *   *特点*：基于内存存储，读写速度极快，支持每秒数千次吞吐（TPS）。

### 8.2 SQLite Relay 深度解析 (The Portable Node)

**8.2.1 数据库模式 (Schema Design)**
源码中定义了两个核心表，这是为了在 SQL 关系型数据库中模拟 Nostr 的 NoSQL 结构：
1.  **`events` 表**：存储消息本体。
    *   `id` (主键): Event ID (Hex)。
    *   `content`: 加密后的 Base64 数据。
    *   `tags`: JSON 格式存储的标签。
2.  **`index_map` 表**：**倒排索引 (Inverted Index)**。
    *   这是查询速度的关键。Nostr 的查询通常是基于 Tag 的（如查询所有发给群 G 的消息）。
    *   结构：`filter_key` (如 `p:Pubkey` 或 `g:GroupID`) -> `event_id`。
    *   *机制*：当一条消息入库时，Relay 会解析其 Tags，将每一个索引项插入此表。查询时，先查此表拿到 ID，再回查 `events` 表。

**8.2.2 并发处理 (WAL Mode)**
在 `init()` 函数中，代码执行了 `PRAGMA journal_mode=WAL;`。
*   **预写日志 (Write-Ahead Logging)**：这允许多个读取者和一个写入者同时操作数据库，极大地提升了聊天场景下的并发性能，防止数据库锁死（Database Locked）。

### 8.3 Redis Relay 深度解析 (The Performance Beast)

Redis 版放弃了 SQL 的复杂查询能力，换取了极致的速度。

**8.3.1 数据结构设计**
*   **哈希 (Hash)**: `event:{id}` -> 存储完整的 JSON 字符串。
*   **有序集合 (Sorted Set / ZSet)**: 用于索引。
    *   `idx:kind:{k}`: 按类型索引。Score = 时间戳。
    *   `idx:author:{pk}`: 按发送者索引。
    *   `idx:tag:{char}:{val}`: 通用标签索引（如 `idx:tag:g:GroupID`）。
    *   *优势*：ZSet 天然支持按时间排序和范围查询（Pagination），这完美契合聊天记录“加载更多”的需求。

**8.3.2 管道操作 (Pipeline)**
源码中的 `save_event` 使用了 `redis.pipeline()`。
*   **原子性**：将保存 Event、更新 Kind 索引、更新 Author 索引、更新 Tag 索引等 5-6 个操作打包成一个网络包发给 Redis。
*   **性能**：大幅减少了网络往返时间（RTT），保证了即使在高负载下，写入操作也是微秒级的。

### 8.4 哑逻辑核心：只做验证，不做裁判

无论是哪个版本，Relay 的核心逻辑都严格遵循 **NIP-01** 和 **NIP-13**。

**8.4.1 签名验证**
Relay 不关心你是谁，只关心数学。它会验证 `sig` 是否由 `pubkey` 对 `id` 进行 Schnorr 签名生成。这保证了**不可伪造性**。

**8.4.2 PoW 验证 (`verify_pow`)**
这是反垃圾的第一道防线。
*   逻辑：`int(event_id, 16) < target`。
*   **拒收策略**：如果消息声称发往官方大厅 (`g:OFFICIAL_ID`) 但难度未达标（Difficulty < 16），Relay 会直接返回 `OK: False` 并丢弃连接。这在源头上清洗了垃圾流量。

**8.4.3 广播机制 (`broadcast_event`)**
当收到新消息时，Relay 会遍历当前所有在线的 WebSocket 连接。
*   **匹配**：检查新消息是否符合连接 A 的订阅过滤器（Filter）。
*   **推送**：如果匹配，立即通过 WebSocket 推送。
*   *注意*：Relay 不知道这个连接背后是 Alice 还是 Bob，它只知道“这个连接订阅了群 G 的消息”。

 **8.4.4 状态自愈 (NIP-33)**
Relay 并非只进不出的垃圾桶。对于特定类型（Kind 30000-39999）的**参数化可替换事件**（如群资料更新），Relay 具备“喜新厌旧”的能力。
*   **逻辑**：当收到新的群名修改请求时，Relay 会自动索引并**物理删除**旧的群名记录。
*   **价值**：这保证了新上线的客户端只需拉取一条数据就能获得最新状态，而不是拉取过去一年的 1000 次改名记录。这是去中心化系统保持轻量高效的关键。

### 8.5 数据生命周期 (Data Lifecycle & TTL)

**8.5.1 临时性哲学**
DageChat 的 Relay 不是档案馆，而是流动站。
*   **配置**：`dagechat-relay.json` 中的 `data_ttl` (默认 30 天)。
*   **清理任务**：后台协程 `ttl_task` 会定期运行。
*   **物理删除**：
    *   SQLite: `DELETE FROM events WHERE created_at < cutoff`。
    *   Redis: `EXPIRE` 命令让 Key 自动消亡。
*   **安全意义**：这是对前向保密（PFS）的补充。即使 Relay 被黑客攻破并脱库，黑客也只能拿到最近 30 天的加密乱码，拿不到一年前的数据。

## 本章问答 (Q&A)

**Q1: 搭建一个 Relay 需要很高的配置吗？**
**A:** 不需要。
*   **SQLite 版**：一台 1核 512MB 内存的 VPS，甚至家里的旧安卓手机（安装 Termux）都能跑。它只需要 Python 环境。
*   **Redis 版**：取决于用户量。对于几千人的社区，一台 2核 4G 的服务器配合 Redis 足以此生。

**Q2: Relay 能看到我的私聊内容吗？**
**A:** **绝对不能。**
Relay 存储的是经过 **XChaCha20-Poly1305** 加密的二进制数据（Base64 编码）。没有私钥，这些数据就是一堆随机噪音。对于 NIP-59 私聊，Relay 甚至不知道这条消息是发给谁的。

**Q3: 既然 Relay 是开源的，如果有人修改代码记录我的 IP 怎么办？**
**A:** 这是一个切实的威胁。
默认情况下，Relay 确实能看到连接者的 IP。恶意的 Relay 管理员可以修改代码记录 IP 日志。
**防御**：
1.  **信任**：只连接你信任的朋友搭建的 Relay。
2.  **技术**： **配置 SOCKS5 代理**。DageChat 客户端支持通用的 SOCKS5 代理协议（在 `relays.json` 中配置 `proxy` 字段）。用户需自行运行 Tor 客户端或 VPN，并将 DageChat 流量指向该端口，从而隐藏真实 IP。

**Q4: 我可以同时连接 SQLite 版和 Redis 版的 Relay 吗？**
**A:** 可以。客户端对 Relay 的实现细节是无感知的。只要它们都遵循 Nostr 协议标准，客户端就可以混合连接，实现数据的多重冗余备份。

**Q5: 如果 Relay 突然关机了，我的消息会丢吗？**
**A:**
*   **已接收的消息**：已经存在你本地硬盘里了，不会丢。
*   **发送中的消息**：如果 Relay 挂了，客户端会尝试重连或发送给列表中的其他 Relay。只要你配置了多个 Relay，消息总能发出去。

**Q6: 为什么 Redis 版比 SQLite 版快？**
**A:** SQLite 是基于磁盘文件的数据库，受限于硬盘 I/O（读写速度）。Redis 是基于内存的数据库，所有数据都在 RAM 中，读写速度是纳秒级的。但 Redis 的成本更高（内存比硬盘贵）。

**Q7: 我能用 DageChat 的 Relay 去连接 Damus 或其他 Nostr 客户端吗？**
**A:** **部分可以。**
我们的 Relay 是标准兼容的（NIP-01）。普通的 Nostr 客户端可以连接上来。
**但是**：
1.  它们看不懂 DageChat 的加密消息（乱码）。
2.  它们可能无法通过 DageChat 的 PoW 校验（如果我们开启了强制高难度），导致发消息失败。

**Q8: Relay 会保存图片文件吗？**
**A:** DageChat 目前的设计是将图片转为 Base64 嵌入 JSON 文本中。所以 Relay 保存的是**巨大的文本字符串**。
*   *优点*：无需额外的文件存储服务（OSS），去中心化更彻底。
*   *缺点*：会占用 Relay 较多的数据库空间。这就是为什么要设置 TTL（过期删除）的原因。

**Q9: 如果我想运营一个收费的 Relay，支持吗？**
**A:** 目前 DageChat 开源版本未内置 NIP-42（认证与付费）逻辑。目的是保持纯粹和低门槛。但代码是开源的，你可以基于 `fastapi` 轻松添加鉴权中间件来实现付费准入。

**Q10: 什么是“基础设施主权”？**
**A:** 意思是你不需要求助于科技巨头来获得通讯能力。
在 Web2 时代，你无法搭建一个“微信服务器”。但在 DageChat，代码在你手中，硬件（哪怕是树莓派）在你家中，你就是网络的一部分。没有人能关闭你口袋里的服务器。
